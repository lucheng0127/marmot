# 6. 风险分析

## 6.1 eBPF 内核版本兼容风险

| 维度 | 详情 |
|------|------|
| **风险等级** | 🔴 高 |
| **描述** | eBPF 程序依赖内核版本特性。TC BPF 需要 5.10+ 内核支持 CO-RE；某些路由器嵌入式内核可能较旧（如 OpenWrt 21.x 使用 5.4 内核） |
| **影响范围** | 在老旧内核设备上无法运行 |
| **概率** | 中（目标设备为现代 Linux 网关，通常使用较新内核） |
| **缓解措施** | ① 最低要求文档化：Kernel ≥ 5.10 ② BPF 程序兼容性测试套件 ③ 提供内核版本检查启动脚本 ④ 可选 legacy mode（fallback 到 iptables）|
| **监控指标** | `/sys/kernel/btf/vmlinux` 是否存在、`uname -r` 版本号 |
| **回退方案** | 在「eBPF 不可用」场景下回退到 iptables TPROXY 模式（降低性能但保证可用性） |

## 6.2 sing-box 集成复杂度风险

| 维度 | 详情 |
|------|------|
| **风险等级** | 🟡 中 |
| **描述** | sing-box 作为 library 集成时存在以下挑战：① `box.New()` API 不稳定，版本升级可能 break ② `option.Options` 结构体庞大复杂 ③ sing-box 内部启动时也会尝试创建 Inbound，需要屏蔽 |
| **影响范围** | 开发阶段重复返工、版本升级适配 |
| **概率** | 高 |
| **缓解措施** | ① 固定 sing-box 大版本，lock 依赖 ② 抽象 Proxy Engine 接口层，隔离直接依赖 ③ 编写 sing-box 集成适配器（adapter pattern）④ 必要的 vendor 机制 |
| **架构建议** | 创建 `pkg/proxy/engine.go` 封装层，对外提供简洁接口：`Engine.Start()`, `Engine.Stop()`, `Engine.Dial(address) → net.Conn` |

---

## 6.3 DNS 污染绕过失败风险

| 维度 | 详情 |
|------|------|
| **风险等级** | 🟡 中 |
| **描述** | 国内 DNS 返回虚假 IP（DNS 缓存投毒）；国外 DNS 可能被 SNI 阻断；GFW 可能对 DoH/DoT 端点进行 DPI 识别 |
| **影响范围** | 部分域名解析错误，导致无法正常访问 |
| **概率** | 中 |
| **缓解措施** | ① DNS 反污染层：检测响应 IP 是否被污染（如对比多个 DNS 结果）② 支持 DNS over HTTPS with ECH (Encrypted Client Hello) ③ 支持自定义 DNS 上游 ④ DNS 结果缓存 + 过期主动刷新 ⑤ 支持 DNS-over-QUIC（未来扩展）|
| **验证方法** | 在真实网络环境下测试 `google.com` 等域名解析是否返回正确地址 |

## 6.4 性能瓶颈风险

| 维度 | 详情 |
|------|------|
| **风险等级** | 🟡 中 |
| **描述** | ① eBPF 自身开销极小，但 TProxy socket 处理是高成本操作 ② sing-box outbound 协议加解密是 CPU 密集型 ③ DNS 查询在高 QPS 下可能成为瓶颈 |
| **影响范围** | 网关设备在高并发时吞吐下降、延迟增加 |
| **概率** | 低（目标场景为家庭/小型办公网关，流量适中） |
| **缓解措施** | ① 核心路径零拷贝设计 ② sing-box 连接复用 ③ DNS LRU 缓存 ④ 连接数统计与限流 ⑤ CPU profiling 工具链（pprof）|
| **基准指标** | 目标：1000 并发 TCP 连接，延迟增加 < 5ms |

---

## 6.5 UDP 处理风险

| 维度 | 详情 |
|------|------|
| **风险等级** | 🟠 中高 |
| **描述** | UDP 透明代理是公认的难点：① UDP 无连接状态，难以关联请求与响应 ② QUIC/HTTP3 使用 UDP，绕过代理导致降级 ③ 大量游戏的 UDP 流量对延迟极度敏感 ④ conntrack 表项容易溢出 |
| **影响范围** | QUIC 连接降级到 HTTP/2、游戏连接受影响、UDP 应用无法代理 |
| **概率** | 高 |
| **缓解措施** | ① sing-box 的 UDP over TCP 模式兜底 ② 支持 QUIC 阻断（可选）③ 精细化 conntrack 超时配置 ④ UDP 直连白名单 ⑤ 分阶段交付：Phase 1 仅 TCP，Phase 2 加入 UDP |

---

## 6.6 规则数据热更新风险

| 维度 | 详情 |
|------|------|
| **风险等级** | 🟢 低 |
| **描述** | GeoIP/GeoSite 数据在运行时热更新失败可能导致路由错误 |
| **影响范围** | 流量分流错误（如国内流量走代理） |
| **概率** | 低 |
| **缓解措施** | ① 原子替换机制（写时复制）② 更新失败使用旧数据兜底 ③ 版本校验 ④ 定时自动更新（每 24h）|

## 6.8 ⭐ v0.3 新增：Decision Cache 重启丢失风险（Phase 2 更新）

| 维度 | 详情 |
|------|------|
| **风险等级** | 🟡 中 |
| **描述** | Phase 2 用户态 Decision Cache 在进程重启后丢失；多 goroutine 并发写入可能导致短暂不一致 |
| **影响范围** | 重启后缓存 warmup 期全部走 Complete Decision Path |
| **概率** | 低 |
| **缓解措施** | ① 重启后缓存自动 warmup（首次访问重新决策，结果一致）② LRU 淘汰防溢出 ③ Phase 5 引入 eBPF Flow Map（BPF FS pin 持久化）后可缓解 |
| **监控指标** | Cache Hit Rate、Cache 条目数、GC 每次清理数 |

## 6.9 风险预防总结（v0.3 更新）

| 阶段 | 应采取的行动 |
|------|-------------|
| 开发期 | 每次 CI 构建检查内核版本兼容性 + Flow Cache 单元测试 + sing-box API 兼容性测试 |
| 测试期 | 建立真实网络环境的端到端测试（国内/国外/DNS）；Conntrack GC 压力测试 |
| 部署期 | 提供 `marmot doctor` 命令诊断环境；检查 BPF Map 上限是否满足预期流量 |
| 运维期 | 收集 Flow Cache 命中率 + BPF Map 占用率指标；建立告警机制 |
